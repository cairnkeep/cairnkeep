<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| sync script → live OpenCode config tree | script renders `@@INFRA_ROOT@@` and installs plugin files into `~/.config/opencode/plugins` |
| rendered INFRA_ROOT → executed server path | the rendered absolute path becomes the `node` target the plugins invoke |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-04-03 | Elevation of Privilege | `@@INFRA_ROOT@@` rendered from an untrusted value → plugins execute an attacker-controlled `node` target | medium | mitigate | Render only from `$ROOT_DIR` computed by the script from its own location (`cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd`), exactly as `sync-claude-assets.sh` does; never take the render value from env or CLI input |
| T-04-10 | Tampering | install writing outside the intended live-root | low | mitigate | Preserve the existing `$LIVE_ROOT/$rel` join and `install -m 0644`; `--live-root` stays operator-supplied and local, matching the existing script contract |
| T-04-04 | Denial of Service | fail-open plugins masking a broken install | low | accept | D-03: plugins no-op on a bad server path rather than wedging a session; the OCP-05 acceptance gate is what confirms the install actually surfaces memory |
| T-04-SC | Tampering | `@opencode-ai/plugin` (SUS scanner verdict) | high | mitigate | No package install — plugins are loose type-only files resolved by OpenCode's runtime loader; a blocking-human checkpoint verifying `npmjs.com/package/@opencode-ai/plugin` is required only if an executor introduces a `package.json` install |
</threat_model>
