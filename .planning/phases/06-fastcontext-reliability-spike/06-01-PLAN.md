<objective>
Build the committed, re-runnable FastContext reliability probe — `scripts/verify-fastcontext-reliability.sh` — that Phase 6's go/no-go verdict is measured with. Per D-01 it mirrors the known-good `scripts/verify-opencode-live-parity.sh` pattern: staged, env-driven, loopback-safe, generous timeouts, single exit-code pass/fail. Per D-03 the probe drives the raw `llama-server` `/v1/chat/completions` endpoint directly with FastContext's own read/glob/grep tool schemas, isolating the variable under test (model + chat-template + `--jinja` + quant) from token-miser's Rust execution loop.

This plan produces ONLY the instrument and its offline self-test. It does not run against a live model — that is the operator-gated Plan 06-02. Every task here is verified by `--self-test`, which exercises the record-and-check + verdict logic against canned fixture responses with no live endpoint (the Nyquist automated backstop required by 06-VALIDATION.md).

Purpose: give the operator a cheap, repeatable probe so the verdict can be re-established whenever the pinned quant/build/template changes (D-01 rationale), instead of a one-off investigation that must be re-derived next time.
Output: `scripts/verify-fastcontext-reliability.sh` with `--self-test`, `--props-only`, `--full`, and `-h/--help`.
</objective>

## D-05 scoring — refined in-session by the operator ("Evidence, not hard gate")

RESEARCH finding #1/#4 established (source-verified against llama.cpp master + a live Hugging Face Hub metadata fetch of the exact pinned quant) that `chat_template_tool_use` will very likely be **ABSENT** from `GET /props` for the FastContext GGUF — by architecture (a single unified Qwen3-family ChatML template with an inline `{%- if tools %}` / `<tool_call>` XML block), NOT by misconfiguration. Read literally, D-05 gate #1 would then force a NO-GO even if the model invokes tools reliably every turn.

**The operator resolved this in-session (chose "Evidence, not hard gate"):**
- The probe STILL queries `GET /props` and records `chat_template`, `chat_template_tool_use` (present-or-absent), `chat_template_caps`, and `build_info` verbatim — this satisfies ROADMAP SC#2 ("the probe checks /props").
- The go/no-go VERDICT is anchored to the empirical per-turn **gate #2** (≥5 prompts × ≥3 turns, every turn `finish_reason == "tool_calls"` WITH a well-formed tool call).
- Absence of `chat_template_tool_use` is a **recorded caveat**, NOT an automatic blocker. The verdict logic MUST NOT auto-NO-GO solely because the field is absent.

This plan implements that scoring in the script (Task 3). Plan 06-02's checkpoint asks the operator to confirm the recorded rubric matches this in-session decision before the verdict is finalized.
