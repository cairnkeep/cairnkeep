## 3. DEFAULT broad query set — documented D-01 reliability finding (NOT a headline)

The harness's default Phase-6-style broad query set (the 5 queries in the recipe table above) was also run live end-to-end. The native side reproduced the harness's deterministic offline numbers exactly:

| # | pattern | native bytes | native hits |
|---|---------|-------------:|------------:|
| 1 | `containment` | 38432 | 12 |
| 2 | `AgentFS.*scope` | 43555 | 12 |
| 3 | `git-provider` | 23034 | 12 |
| 4 | `memory-wakeup` | 44274 | 12 |
| 5 | `infraRoot` | 46004 | 12 |
| **Total** | | **195299 bytes** | |

The explore side of this broad set was **unreliable** and every citation is labeled clearly below as non-existent/hallucinated (none of these paths exist in cairnkeep — verified against the repo tree):

1. "Where is scope path containment implemented in this repo?" → 0 citations after 14 turns / 31 tool_calls, took 125s — **EXCEEDS** the harness's own 120s per-query timeout (this is exactly the scenario the harness's `--full` fail-loud behavior exists for, per D-04). citation-note bytes=45.
2. "Where is the AgentFS project scope resolved in this repo?" → returned `agentfs/README.md:1-100`, `agentfs/src:1-100` — **HALLUCINATED, non-existent path** (no `agentfs/` directory exists in cairnkeep). bytes=41, 5s.
3. "Where is the git-provider abstraction configured in this repo?" → returned `git2-rs/src/lib.rs:1-1000` — **HALLUCINATED, non-existent path** (no `git2-rs/` in cairnkeep; the `:1-1000` range is a fabricated round number). bytes=25, 4s.
4. "Where does the OpenCode memory-wakeup plugin live in this repo?" → returned `opencode/plugins/memory-wakeup:1-100` — **HALLUCINATED, non-existent path** (cairnkeep has an `opencode/` directory but not that subpath). bytes=36, 4s.
5. "Where do the asset-sync scripts render the infra-root placeholder value?" → returned 4 citations under a fabricated `infrastructure/scripts/...` prefix — **HALLUCINATED PREFIX** (the real files live at `scripts/`, cairnkeep has no `infrastructure/` directory). bytes=243, 7s.

A naive byte-delta computation on this broad set would read `195299 → ~390 bytes` (~99.8% "savings"). **This number is explicitly NOT the CTX-07 headline** — it is built entirely on empty or hallucinated citations that point to files which do not exist in this repo. It is recorded here only as a transparent D-01 model-reliability finding: on broad/loosely-worded queries this 4B model tends to wander to the turn cap (query 1) or fabricate plausible-sounding but non-existent paths (queries 2-5).

## 4. D-03 verdict

**PASS** — on the verified tight queries (Section 2): net byte savings are strongly positive (99.9%+) with **correct, independently-verified** explore output. This is the number CTX-07 is closed against.

The DEFAULT broad query set (Section 3) is recorded as a **documented D-01 model-reliability gap**, never a silent pass — its naive ~99.8% figure is explicitly disqualified above because the underlying citations are hallucinated or empty.

Cairnkeep's ~99.9% pinpoint-query magnitude need not match FastContext's own paper figure of ~60% (D-03) — the two are not the same measurement shape: cairnkeep's native "before" side is a full grep-dump across 12 windows (a generous, conservative floor per Section 1), while explore's "after" side is a single pinpoint citation. The magnitude difference is expected and does not affect the verdict.

The reliability caveat in Section 3 does **not** block milestone close-out (D-04) — it is recorded transparently as a forward-looking finding, consistent with the Phase 6 spike precedent (06-SPIKE.md) of recording model-reliability caveats without gating verdicts on them.
